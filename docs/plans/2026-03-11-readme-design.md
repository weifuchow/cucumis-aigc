# README Design

**Date:** 2026-03-11

## Purpose

将 `README.md` 从偏“项目介绍文”重构为面向内部团队与创始团队的项目总说明书，用于统一以下认知：

- `cucumis-aigc` 的项目定位与问题定义
- 系统架构与核心设计原则
- 视频生产流程与模块职责
- v1 边界、阶段重点与后续路线图

## Audience

- 内部团队成员
- 创始团队

这份文档不以外部宣传为主要目标，也不承担开发手册或 API 文档职责。

## Writing Direction

- 高密度，但避免写成实现细节手册
- 先业务与问题，再架构与流程，最后阶段边界与路线图
- 使用明确判断句，减少模糊愿景表述
- 保留核心技术概念：`Orchestrator`、`Skill`、`Filesystem`、`Timeline Schema`、`FFmpeg`、`Observer`

## Target Structure

1. 项目定位
2. 为什么要做
3. 项目目标
4. 核心架构
5. 视频生产流程
6. 核心设计原则
7. v1 边界
8. 路线图
9. 长期价值

## Content Decisions

### 1. 项目定位

明确项目是“面向短视频生产的本地化 AIGC 编排系统”，强调它不是单点生成工具，也不是纯 SaaS 平台。

### 2. 问题定义

聚焦真实生产断点：

- 流程分散
- 中间产物不可见
- 风格一致性差
- 渲染与复用成本高
- 自动化链路不稳定

### 3. 目标边界

README 重点说明 v1 要建立：

- 本地单机闭环
- 统一 Skill 体系
- 可追踪、可恢复的工作流
- 基于 Timeline + FFmpeg 的稳定渲染出口

### 4. 架构表达

采用上位结构表达：

`Orchestrator -> Skills -> Filesystem -> Timeline Schema -> FFmpeg -> Observer`

并说明各层职责，不下沉到接口级设计。

### 5. 流程表达

保留已有 12 步视频生产流程，但压缩冗余说明，强化每一步的产出与上下游关系。

### 6. 原则与边界

新增“v1 不做什么”，避免团队将 README 理解为无限扩张的愿景清单。

### 7. 路线图

分为 `v1 / v2 / v3` 三阶段，强调当前阶段关注点与后续扩展方向。

## Non-Goals

- 不写安装教程
- 不写命令级运行说明
- 不写模块级开发细节
- 不写对外宣传口径

## Constraints

- 当前工作区不是 Git 仓库，无法按流程提交设计文档 commit
- 仓库内当前只有 `README.md`，文档需避免虚构不存在的实现细节
